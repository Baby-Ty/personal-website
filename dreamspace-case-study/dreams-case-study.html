Planning

From brief to architecture

Netsurit needed a simple way for employees to set goals, track progress, and stay connected with their team.

The goal was a lightweight system that could run cheaply on the existing Microsoft stack.

The sections below walk through the build and the Claude workflows used to develop it.

Product requirements
  • Single sign-on with Microsoft accounts
  • Weekly goal-setting and reflection workflow
  • Coach and team management
  • Microsoft Teams meeting integration
Platform constraints
  • Hosted entirely on Azure
  • Authentication through Entra ID
  • Microsoft Graph for calendar and identity
  • Cost target of ~$2 per user per year
Stack
  • Azure Functions · Static Web Apps
  • Cosmos DB · Blob Storage
  • Entra ID · Microsoft Graph
  • Azure AI Foundry · Claude Code
Setup

Claude project setup

The platform was built using Claude Code, which meant most of the work happened through prompt sessions.

To avoid repeating the same context every session, a simple structure was built so each session started from the same baseline.

Three pieces ended up forming the setup:

  • CLAUDE.md — project context and conventions
  • skills/ — reusable workflows
  • agents/ — specialist contexts
None of this existed at the start. It evolved over time.

CLAUDE.md — project context

CLAUDE.md was kept short.

It ended up containing only the essentials:

  • architecture overview
  • project structure
  • development conventions
  • common commands

Its purpose was simply to give each session enough context to stay aligned with the project.

Reusable workflows were defined as skills, while more focused tasks were handled through agents.

CLAUDE.md
## Dreams Program
React + Vite frontend, Azure Functions backend, Cosmos DB.
Auth via MSAL / Entra ID. AI via Azure AI Foundry.

## Architecture
Pages → Layouts → Hooks → Services. No fetch in components.
Files under 400 lines. Reference src/pages/dream-team/.

## API
Auth middleware on every endpoint (api/utils/authMiddleware.js).
Verify userId ownership. Return ok() or fail() — never raw JSON.

## Cosmos DB
Always query with partition key. 10 containers:
users, dreams, currentWeek, pastWeeks, teams, scoring,
connects, meeting_attendance, prompts, coaching_alerts.

## Common Commands
npm run dev              # frontend localhost:5173
cd api && func start     # backend localhost:7071
npm test                 # run tests
npm run docker:start     # local Cosmos emulator
npm run setup:verify     # verify local DB setup

## Data Scripts
Any script that touches live data must have a dry-run
mode by default. Log what would change, write nothing.
Require --execute to perform real writes.
Test on a single user with --test-user before bulk runs.

Skills — reusable workflows

Skills were added as patterns repeated across sessions.

Common instructions were moved into skills instead of being written in prompts every time.

.claude/skills/

Each skill includes a short description telling Claude when to apply it.

new-feature/SKILL.md
---
name: new-feature
description: Building a new page or feature.
  Apply when asked to add a new section, page,
  or user-facing capability to the app.
---
Create all three layers using src/pages/dream-team/
as the reference pattern:

1. Page + layout in src/pages/
2. Hook in src/hooks/ — data and business logic
3. Service in src/services/ — API calls only
4. Azure Function in api/ — auth middleware,
   userId check, ok/fail response

Wire all layers together before finishing.
plan/SKILL.md
---
name: plan
description: Planning a new feature or change
  before writing code. Apply when asked to build
  something non-trivial or when scope is unclear.
---
Before writing any code:

1. Identify all layers affected
   - Which pages, hooks, services change
   - Which API endpoints are new or modified
   - Which Cosmos containers are read or written
2. List files that will be created or changed
3. Identify risks
   - Data shape changes that affect existing records
   - Auth or permission edge cases
   - Dependencies between the new and existing code
4. Confirm the plan before starting

Do not write code until the plan is agreed.
new-endpoint/SKILL.md
---
name: new-endpoint
description: Adding an Azure Function API endpoint.
  Apply when asked to create, add, or build a
  new backend endpoint or API route.
---
Use api/getUserData/index.js as the reference.

- Import and apply authMiddleware
- Verify req.userId matches the resource owner
- Wrap response in ok() or fail() from
  api/utils/responseHelpers.js
- Register in api/host.json if needed
onboard/SKILL.md
---
name: onboard
description: Getting oriented in the codebase.
  Apply at the start of a new session, when
  context has been cleared, or when asked to
  understand the project before making changes.
---
Read the following in order:

1. CLAUDE.md — architecture and conventions
2. src/pages/ — list of pages and their purpose
3. src/hooks/ — data and state layer
4. src/services/ — API call layer
5. api/ — backend endpoints
6. tailwind.config.js + src/index.css — design tokens

Then summarise:
- What the app does
- The current state of the area being worked on
- Any obvious issues in scope before starting

Do not make changes during onboarding.
review/SKILL.md
---
name: review
description: Reviewing code before committing or
  deploying. Apply when changes are ready to push,
  before a PR, or before any production deployment.
---
Check the following before marking complete:

- No file exceeds 400 lines — split if needed
- Every new endpoint has authMiddleware applied
- Every endpoint verifies userId ownership
- All new inputs validated with Zod before use
- Responses use ok() or fail() — no raw JSON
- No hardcoded credentials, IDs, or env values
- No console.log left in production paths
- Cosmos queries include the partition key

Return a list of findings or confirm it's clean.
debug/SKILL.md
---
name: debug
description: Diagnosing a bug or unexpected
  behaviour. Apply when investigating any issue
  before suggesting a fix.
---
Trace through all layers before proposing a fix:

1. Component — what state or props trigger it
2. Hook (src/hooks/) — how data is loaded
3. Service (src/services/) — what the API sends
4. Azure Function (api/) — endpoint behaviour
5. Cosmos DB — what is actually stored

Identify the exact layer, then fix it there.

Agents — task-specific contexts

Agents came later.

Certain areas like Azure infrastructure, Microsoft Graph integrations, and security reviews kept producing answers that needed correcting. The problem wasn't capability, it was context.

Agents handle this by running those tasks in isolated contexts with their own instructions and tools.

Agents live in:

.claude/agents/

Each agent loads the context needed for that domain before the task begins.

Examples include:

  • azure-infra — infrastructure and deployment changes
  • security — endpoint and authentication reviews
  • graph-api — Microsoft Graph integrations

This keeps the main session focused while specialised tasks run with the context they need.

azure-infra/AGENT.md
---
name: azure-infra
description: Azure infrastructure expert. Use
  proactively when reviewing or changing Azure
  Functions, Cosmos DB, Blob Storage, Static Web
  Apps config, or CI/CD pipelines.
tools: Read, Glob, Grep, Bash
model: sonnet
---
You are a senior Azure architect. This project runs
on Azure Static Web Apps, Azure Functions (Node.js),
Cosmos DB with 10 containers, and Azure Blob Storage.
Auth is via Entra ID / MSAL.

When reviewing infrastructure changes:
- Flag any Cosmos DB queries missing a partition key
- Check Function bindings and host.json config
- Verify environment variables are in Azure Portal,
  not hardcoded or in committed config files
- Review autoscale settings and RU/s implications
- Check CORS, auth config, and Static Web App routes
- Flag anything that would increase monthly cost
  beyond the ~$995/year baseline

Reference: README.md for the full architecture overview.
security/AGENT.md
---
name: security
description: Web app security expert. Use
  proactively after adding endpoints, changing
  auth logic, or before any production deployment.
tools: Read, Glob, Grep
model: sonnet
---
You are a senior web application security engineer.
This app uses MSAL / Entra ID for auth, Azure
Functions for the backend, and Cosmos DB partitioned
by userId.

Review for:
- Endpoints missing authMiddleware from
  api/utils/authMiddleware.js
- Missing userId ownership checks — authenticated
  user must match the resource being accessed
- Hardcoded credentials, API keys, or secrets
  in any committed file
- CORS misconfiguration in staticwebapp.config.json
- Input validation gaps — all user input must pass
  through Zod schemas before reaching the database
- Any operation that could allow cross-user data
  access by manipulating request parameters

Return a risk-ranked list of findings with
the exact file and line for each.
graph-api/AGENT.md
---
name: graph-api
description: Microsoft Graph API specialist.
  Use when integrating calendar, mailbox settings,
  user profile, or any Graph endpoint. Invoke before
  writing new Graph calls or debugging auth failures.
tools: Read, Glob, Grep, WebFetch
model: sonnet
---
You are a Microsoft Graph API integration specialist.

This app authenticates via MSAL with Entra ID.
Tokens are acquired on the frontend and forwarded
to Azure Functions which call Graph on behalf of
the user (on-behalf-of flow).

Current integrations:
- Calendar events (read/write) via /me/events
- Mailbox settings for timezone detection
- User profile data for display and identity

When writing new Graph calls:
- Confirm the correct permission scope is declared
  in the Entra app registration (delegated vs app)
- Use the correct base URL (graph.microsoft.com/v1.0)
- Verify the OData query parameters for the endpoint
- Check token forwarding in api/utils/authMiddleware.js
- Never hardcode Graph tokens or user IDs

When debugging:
- Check the token has the required scope
- Confirm the Entra app has admin consent for the scope
- Test with Graph Explorer before assuming code error

Return the exact Graph endpoint, required permissions,
and a working code example using this project's
existing auth pattern.
None of this was set up on day one. CLAUDE.md started as a few lines and grew when sessions drifted. Skills appeared when the same instructions kept getting typed. Agents arrived when specialist questions kept needing correction. The configuration layer is what the project accumulated — and once it was committed, every session after that started from the same place.
Workflow

How the prompts worked

The sessions that produced the best results followed a simple structure: context → tasks → constraints.

Clear context and specific instructions reduced the amount of rework needed later.

Full-stack feature from a single session
The Dream Team page needs meeting attendance tracking.

- Cosmos schema for meeting_attendance, partitioned by teamId
- Azure Functions: getMeetingAttendance, saveMeetingAttendance
- Schedule team meetings using the Graph Calendar API
- Attendance checklist card — present/absent toggles
- Attendance rate as a metric in the Report Builder

Coaches schedule and update meetings. Team members can view meetings and attendance.
Schema, backend, Graph API integration, and UI — all connected in one session.
Production incident — 35 users on the wrong week
35 users were rolled over to the next week twice.
A local development instance ran the rollover at the same time as the Azure timer.
Affected users are on W08, but should be W07.

Backups are in backups/.

- Restore currentWeek and pastWeeks from backups for all users
- Fix the timer so local development cannot trigger production jobs
- Add a safeguard so the rollover can only run once per week
Rollback script written, tested in dry-run, executed, and the root cause fixed in the same session.
Cross-system integration — Graph API
Two things to fix with meeting scheduling:

Meetings are created in the wrong timezone — the coach's timezone isn't being read from Graph.
Meetings aren't Teams-enabled, so no join link is generated.

Fetch the coach's timezone from mailboxSettings when the scheduling form loads.
Add a timezone dropdown that defaults to the coach's timezone.
Set isOnlineMeeting: true in the Graph event.
Touching the Graph API, Cosmos, and the React form simultaneously. Claude looked up the correct Graph payload fields mid-session.
Honest Assessment

Where it struggled

Not everything worked the first time, but most issues came down to context, not capability.

Infrastructure context
Early Azure infrastructure questions produced sensible answers, but not answers specific to this architecture. Once the azure-infra agent loaded the project context, infrastructure sessions became far more reliable.
Less familiar APIs
For common stacks like React, Azure Functions, and Cosmos DB the first pass was usually correct. Less common integrations like Graph mailbox settings or Entra token flows sometimes needed verification or small tweaks.
Replication

Starting a new project

How I'd start from scratch.

Quick Start
1.  Install Claude Code
    npm install -g @anthropic-ai/claude-code

2.  Initialise the project
    cd your-project && claude
    /init  →  review and trim the generated CLAUDE.md

3.  Add what /init misses to CLAUDE.md
    - Architecture pattern (your layer structure)
    - Key file references (the canonical example for each layer)
    - API conventions (auth pattern, error pattern)
    - Data constraints (partition keys, schema rules)
    - Common commands (dev, test, build, deploy)

4.  Create your skills in .claude/skills/
    - plan/SKILL.md          (think before you code)
    - onboard/SKILL.md       (orient in the codebase)
    - new-feature/SKILL.md   (scaffolds all layers)
    - new-endpoint/SKILL.md  (API boilerplate)
    - review/SKILL.md        (pre-push pattern check)
    - debug/SKILL.md         (full-stack trace before fix)

5.  Create specialist agents in .claude/agents/
    - One per external system (Azure, Graph API, etc.)
    - One security agent for pre-deployment review
    Include real architecture details in each agent's context

6.  Configure .claude/settings.json
    - Set ask for git push
    - Deny .env and secrets files

7.  Set up local dev environment
    - Docker + database emulator for safe local dev
    - Separate local config that never touches production

8.  Session discipline from day one
    - One context window per active workstream
    - /clear or new session after heavy iteration
    - Don't wait for the 200k token limit — move earlier

9.  Commit the .claude/ folder
    The whole team gets the same setup.
    Standards don't drift between sessions.
Takeaway

What actually mattered

Context, not prompting. The sessions that worked best weren't about clever prompts. They worked because Claude had the right project context. CLAUDE.md, skills, and agents are how you give it that context.

Structure compounds. Every skill added meant the same instructions never needed typing again. Every agent meant fewer wrong answers in that domain. The setup investment pays back across every session after.

Scope sessions like tasks. The clearer the context, the tasks, and the constraints — the more reliable the output. One workstream per session. Fresh context over long degraded ones.
